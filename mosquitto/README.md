# CrossChat Mosquitto Reference Setup (WIP)

This is a reference MQTT broker configuration for a 3-server CrossChat deployment.

## Servers

| Username | Password | Server ID | Implementation |
|----------|----------|-----------|----------------|
| `0`      | `pass0`  | `0`       | Python         |
| `1`      | `pass1`  | `1`       | Lua (GMod)     |
| `2`      | `pass2`  | `2`       | C#             |

Passwords match the corresponding `config.X.json` files in the parent directory.

## Files

- `mosquitto.conf` - Main broker config
- `aclfile` - Topic access control rules
- `passwd_plain` - Plaintext credentials
- `passwd` - Hashed credentials (generated by `gen_passwd.sh`)
- `gen_passwd.sh` - Script to hash the password file (requires `mosquitto_passwd`)

## Usage

```bash
# 1. Hash passwords (needs mosquitto package installed):
./gen_passwd.sh

# 2. Start broker:
mosquitto -c mosquitto.conf
```

## ACL Permission Model

### Design Goal

**Prevent a server from impersonating another server.**

The CrossChat protocol uses topics like:
- `crosschat/m/<FROM_SERVER>/<TO_SERVER>/<ENDPOINT>/...`
- `crosschat/state/<SERVER_ID>/<KEY>`

### ACL Rules (`aclfile`)

```
pattern read crosschat/state/+/#
```

**All servers can READ all state topics** - needed for server discovery and presence.

---

```
user 0
topic read crosschat/m/+/0/#
topic write crosschat/state/0/#
topic write crosschat/m/0/+/#
```

Server `0` can:
- **READ** messages addressed TO `0`: `crosschat/m/*/0/*`
- **WRITE** its own state: `crosschat/state/0/*`
- **WRITE** messages FROM `0`: `crosschat/m/0/*/*`

Server `0` **CANNOT**:
- Write to `crosschat/m/1/2/...` (impersonate server 1 sending to 2)
- Write to `crosschat/m/2/0/...` (impersonate server 2 sending to 0)
- Write to `crosschat/state/1/status` (spoof server 1's presence)

**Same pattern applies to users 1 and 2.**

---

## What This Prevents

| Scenario | Result |
|----------|--------|
| Server 0 sends to topic `crosschat/m/1/2/say/5` | ❌ DENIED - `0` cannot write to `m/1/...` |
| Server 0 publishes `crosschat/state/1/status` | ❌ DENIED - `0` cannot write to `state/1/...` |
| Server 1 sends `crosschat/m/1/0/say/3` | ✅ ALLOWED - this is legitimate traffic from 1 to 0 |
| Any server reads `crosschat/state/+/status` | ✅ ALLOWED - needed for server discovery |

## ⚠️ Remaining Spoofing Vulnerabilities

This ACL is **topic-based only**. It does NOT protect against:

### 1. Same-Server User Impersonation

Server `0` controls all topics under `crosschat/m/0/...`. This means:

- Server `0` can publish `crosschat/m/0/2/say/999` where user ID `999` doesn't actually exist on server 0
- Server `0` can send PMs "from user 123" even if user 123 was fabricated
- Server `0` can publish fake `user` add/leave/update messages

**Mitigation**: This is trusted-server architecture. If you don't trust server 0, don't authenticate it to the broker.

### 2. State Spoofing Within Own ID

Server `0` can write anything to `crosschat/state/0/*`. It could:
- Publish fake metadata
- Spoof its own `started` timestamp
- Generally lie about its own state

**Mitigation**: This is inherent to the design - servers must be trusted to report their own state truthfully.

### 3. No Per-Topic Payload Validation

Mosquitto ACLs cannot inspect or validate JSON payloads. A compromised server:
- Could send malformed or malicious JSON
- Could exploit parser bugs in other implementations
- Could send unexpected endpoint types

**Mitigation**: All implementations must validate incoming payloads defensively.

### 4. OOC Messages

Out-of-band messages (`crosschat/m/<from>/<to>/ooc/<type>`) include:
- `aowl_kick`, `aowl_ban`, `aowl_slap` - Remote admin actions
- `lua` - Remote Lua code execution (on Lua servers)

With the current ACL:
- **Server 0 CANNOT send OOC "as server 1"** (topic prefix `m/1/` blocked)
- **But server 0 CAN send any OOC to any server "as itself"** (topic `m/0/...` allowed)

If server 0 is compromised, it can:
- Kick/ban/slap users on any connected server via `aowl_*` OOC
- Execute arbitrary Lua on server 1 via the `lua` OOC endpoint

**This is by design** - CrossChat's remote admin system assumes all authenticated servers are mutually trusted.

**Mitigation**: 
- Only authenticate servers you fully trust
- Lua servers can set `lua_allow_remote 0` to disable remote code execution
- Consider additional network-level access controls for production deployments

### 5. Broadcast / Wildcard Concerns

The pattern `topic read crosschat/state/+/#` allows all servers to:
- See when any server comes online/offline
- Read all server metadata

This is required for CrossChat to function, but means:
- Servers learn about the existence of all other servers
- No "stealth" servers

## Production Hardening Checklist

Before using in production:

1. **Change all passwords** in `passwd_plain` and all `config.X.json`
2. **Consider TLS**: Add TLS listener to `mosquitto.conf` (use MQTT over TLS on port 8883)
3. **Restrict by source IP**: Use Mosquitto's `listener` with `bind_interface` or firewall rules
4. **Use unique passwords per deployment**: Never reuse reference credentials
5. **Isolate the MQTT broker**: Don't expose it to the public internet unless necessary
6. **Set `lua_allow_remote 0`**: On Lua servers if remote code execution isn't needed
